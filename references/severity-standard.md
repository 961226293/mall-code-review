# 严重级别定义

## P0 - 阻塞合并

满足任一情况可判为 P0：

- 明确的安全漏洞
- 数据写错、数据丢失、资金/库存/订单等关键业务正确性问题
- 明确会导致线上故障或严重回归
- 修复成本低但影响极高，必须立即处理

输出要求：

- 必须写清楚触发条件、影响范围、为何阻塞

## P1 - 高优先级

满足任一情况可判为 P1：

- 明确的逻辑错误，但影响范围小于 P0
- 高概率引发生产问题的事务、并发、缓存、外部调用问题
- 严重的架构越层或实现方式错误，若继续积累会明显增加风险

输出要求：

- 默认建议在当前这次修改中修复

## P2 - 中优先级

满足任一情况可判为 P2：

- 可维护性问题较明显
- 代码写法不符合团队标准，且会持续增加后续改动成本
- 有一定风险，但不构成立即阻塞

输出要求：

- 可以在当前修改中修，或明确列为后续跟进

## P3 - 低优先级

满足任一情况可判为 P3：

- 命名、表达、注释、轻微风格问题
- 不影响正确性和主要行为的改进建议

输出要求：

- 不要把纯风格问题拔高成阻塞项

## 分级原则

- 优先按“真实风险和影响”分级，不按“主观不喜欢”分级
- 如果问题只是风格不同，但不影响正确性、可维护性、团队规则或运行风险，不要硬提 finding
- 同一个问题如果同时影响正确性和维护性，以更高风险级别为准
