# 团队规范检查清单

本文件提炼了团队后端规范中明确、可执行、与代码评审直接相关的规则。只在变更代码确实涉及这些点时进行检查，不要机械套用。

## 1. 项目与包结构

- 遵循既有 `mall-xxx-api / mall-xxx-core / mall-xxx-web` 拆分
- 在该系列项目中新增业务包时，保持 `com.sdt.chinamarket...` 前缀风格
- 不要为了省事把 controller、service、mapper、entity 放错模块

## 2. 注释与可维护性

- 类注释应具备
- 对外或重要接口方法应具备注释
- Param / DTO / VO 属于外部或重要业务契约时应具备注释

## 3. 抽象与实现

- 在存在多供应商、多实现可能性的地方，优先接口与实现分离
- 如果业务逻辑被硬编码绑定到某个具体 supplier / client 实现，而抽象已经存在或明显需要，应作为问题指出

## 4. 异常与返回约定

- 优先沿用既有业务异常模式和标准化 code / message 体系
- 如果异常处理写法削弱排障能力、让问题不可追踪，应指出
- 在项目约定适用时，List 返回字段优先返回 `[]` 而不是 `null`

## 5. MySQL / 查询规范

- 如果变更包含表结构或 DO 模式，关注是否缺少通用审计字段
- JSON 类长字符串默认不应收窄到小 varchar
- 如果变更包含 schema，常用查询字段应考虑索引
- 明确指出高风险全表查询；团队规范已明确禁止业务代码中的无边界全表扫描

## 6. Redis 规范

- Redis 不应作为持久化业务数据的真实来源
- 可接受用途包括缓存、分布式锁、session 等可容忍数据丢失场景

## 7. 配置规范

- 稳定不变的本地配置可放在本地 properties
- 易变配置在使用 Nacos 等分布式配置时，不应再硬编码到本地
- 如果代码把运行期易变值写死在本地配置或源码中，应指出环境混淆风险

## 8. 日志规范

- 避免不必要的大 JSON、大结果对象日志
- 在支持参数化日志时，不要使用字符串拼接写日志
- 如果日志风格明显更像调试代码，且落在生产热路径或敏感路径，应指出风险

## 9. 运行安全规则

- 暴露到公网环境的后门/订正接口应有明确安全开关或控制
- 如果本次修改落在关键业务失败路径上，应关注日志或告警可观测性

## 10. CR 过程规则

- 当 diff 改动关键流程时，把冒烟影响、发布风险、影响范围说明一并作为 review 关注点
